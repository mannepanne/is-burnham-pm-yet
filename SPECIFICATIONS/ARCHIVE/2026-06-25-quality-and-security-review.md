## What is in good shape (no action needed)

So the fixes below are read in context — the fundamentals are sound:

- **Secrets management is clean.** `.dev.vars` is gitignored and was never tracked; no API-key prefixes (`pplx-`, `sk-ant-`) appear in any blob in git history; `wrangler.toml` never held inline secrets; the deploy workflow uses `${{ secrets.* }}`. The committed KV namespace id and the Cloudflare Analytics beacon token are non-secret by design.
- **The Worker degrades gracefully** at every stage (empty pool, judge failure, KV absent, API unreachable) with sensible fallbacks, and validates the LLM-returned `verdict` against an allowlist server-side before the client trusts it.
- **The front end** is thoughtfully built — minimum-loading-time UX, responsive breakpoints, `prefers-reduced-motion`, WCAG-AA-tuned colours, and test hooks (`?force=`, `?simulate=`).
- Wikidata data is rendered via `textContent` only (no injection there). Only runtime-adjacent dependency is `wrangler` (dev); lockfile committed.

## Security findings

### S-1 — Stored XSS via unescaped `innerHTML` (Critical, blocking)

**Location:** `public/index.html:1086` (`renderArticles`); same pattern in `renderCannedTrio` (1124) and `renderJudgeFailed` (1168).

**Problem.** Article cards interpolate `outlet`, `date`, `title`, and `caption` directly into `innerHTML` with no escaping. The data is externally controllable end-to-end:

- `outlet` / `title` / `date` come verbatim from the Perplexity result (`src/worker.js:150`), never sanitised. An attacker who publishes a web article whose title contains markup (e.g. `<img src=x onerror=…>`) can get it surfaced into the pool and executed in every visitor's browser.
- `caption` is generated by Claude, but attacker-publishable **full article text** is concatenated into Claude's system prompt (`src/worker.js:278,292`), so a crafted article can prompt-inject the model into emitting markup that then lands in the DOM unescaped.

The malicious card is cached in KV for 6 hours and served to all visitors. There is no auth, so session theft is not the main risk — but defacement, redirect-to-phishing, and drive-by script injection all are.

Note: the `verdict` value interpolated into `class="…"` (lines 1080, 1093) is **not** exploitable, because `verdict` is allowlist-validated server-side (`src/worker.js:305,485`). Treat that as defence-in-depth only.

**Fix.** Stop using `innerHTML` for untrusted data. Preferred — build nodes and assign text via `textContent`:

```js
function renderArticleCard(article) {
  const card = document.createElement('article');
  card.className = `article-card ${article.verdict}`; // verdict is server-validated

  const meta = document.createElement('div');
  meta.className = 'article-meta';
  const outlet = document.createElement('span');
  outlet.className = 'article-outlet';
  outlet.textContent = article.outlet || 'Unknown';
  const date = document.createElement('span');
  date.className = 'article-date';
  date.textContent = article.date || '';
  meta.append(outlet, date);

  const title = document.createElement('h3');
  title.className = 'article-title';
  title.textContent = article.title || 'Untitled';

  const verdict = document.createElement('span');
  verdict.className = 'article-verdict';
  const label = document.createElement('span');
  label.className = `verdict-label verdict-${article.verdict}`;
  label.textContent = getVerdictLabel(article.verdict);
  const caption = document.createElement('span');
  caption.className = 'verdict-caption';
  caption.textContent = article.caption || '';
  verdict.append(label, caption);

  card.append(meta, title, verdict);
  return card;
}
```

Alternatively, add an `escapeHtml()` helper and run every interpolated field through it before it touches `innerHTML`. Migrate `renderCannedTrio` and `renderJudgeFailed` to the same approach so the pattern can't regress (they use static/mock data today, but share the sink).

**Acceptance criteria.**
- [ ] No untrusted field reaches `innerHTML` unescaped in any render path.
- [ ] A test article with `<img src=x onerror=…>` in its title renders as inert text, not an element.
- [ ] `renderArticles`, `renderCannedTrio`, `renderJudgeFailed` all use the safe path.

### S-2 — Missing security headers / no CSP (Medium)

**Location:** no `public/_headers` file; the Worker sets no headers on asset responses.

**Problem.** The project's own standard (`.claude/CLAUDE.md`, `technology-preferences.md`) requires a securityheaders.com **A grade minimum** before production. There is no CSP, which also means no backstop for S-1.

**Fix.** Add `public/_headers` (honoured by Workers static assets):

```
/*
  Content-Security-Policy: default-src 'self'; script-src 'self' https://static.cloudflareinsights.com; style-src 'self' 'unsafe-inline' https://fonts.googleapis.com; font-src https://fonts.gstatic.com; connect-src 'self' https://query.wikidata.org; img-src 'self' data:; frame-ancestors 'none'; base-uri 'none'
  X-Frame-Options: DENY
  X-Content-Type-Options: nosniff
  Referrer-Policy: strict-origin-when-cross-origin
  Permissions-Policy: geolocation=(), microphone=(), camera=()
```

The inline `<script>` (`index.html:743`) and inline `<style>` need a nonce/hash, **or** the script should be extracted to an external file so `script-src` need not include `'unsafe-inline'`. `style-src` keeps `'unsafe-inline'` for now (large inline stylesheet); tighten later if desired. The CSP must allowlist Google Fonts and `static.cloudflareinsights.com`.

**Acceptance criteria.**
- [ ] `public/_headers` present with the headers above.
- [ ] securityheaders.com gives an A (or better) against staging, then production.
- [ ] `script-src` does not rely on `'unsafe-inline'` (script extracted or hashed).

### S-3 — `/api/refresh` secret passed in the query string (Medium)

**Location:** `src/worker.js:374` (`handleRefresh`).

**Problem.** The shared secret travels in `?key=…`, leaking into Cloudflare edge/access logs, the `Referer` header, browser history, and bookmarks. Gating the paid pipeline behind a secret is the right control; query-string transport undermines it.

**Fix.** Read the secret from a request header instead:

```js
async function handleRefresh(env, req) {
  const provided = req.headers.get('x-refresh-key');
  if (!provided || provided !== env.REFRESH_SECRET) {
    return new Response('Unauthorized', { status: 401 });
  }
  // …unchanged…
}
```

Update the `fetch` handler to pass `req` (it currently passes `url`). Rotate `REFRESH_SECRET` after the change since the old value may be in logs. (Constant-time comparison is not worth chasing here — a high-entropy secret over network jitter is not a practical timing oracle; the transport change is the meaningful fix.)

**Acceptance criteria.**
- [ ] Secret read from a header, never the URL.
- [ ] `REFRESH_SECRET` rotated.
- [ ] Manual check: a refresh with the wrong/absent header returns 401.

### S-4 — Cost amplification on empty-result windows (Medium-Low)

**Location:** `src/worker.js:358,386`.

**Problem.** The cache key is fixed (`commentary:v1`), so an attacker cannot force a miss on demand — good. The narrower gap: results are only cached `if (result.articles?.length)`. When the pipeline returns zero articles (Perplexity down or a parse failure), nothing is cached, so every subsequent `/api/commentary` hit re-runs the full paid Perplexity + Claude pipeline until results recover. There is no rate limit and no concurrency guard, so TTL expiry can also trigger a thundering herd of concurrent pipeline runs.

**Fix.**
- Cache a short-TTL negative result (e.g. 60–120s) even when output is empty, so the per-request amplification is broken.
- Add a Cloudflare rate-limiting rule on `/api/commentary`.
- Optionally add a single-flight lock around `runPipeline` (a short-lived KV marker) to prevent concurrent rebuilds.

**Acceptance criteria.**
- [ ] An empty pipeline result is cached briefly (not re-run on every request).
- [ ] Rate-limiting rule applied to `/api/commentary`.

### S-5 — SSRF surface in `fetchArticle()` (Low, defence-in-depth)

**Location:** `src/worker.js:207,242`.

**Problem.** `fetchArticle()` fetches arbitrary URLs returned by Perplexity. Real in shape, but low impact in the Workers runtime: there is no reachable cloud-metadata endpoint, the URL is LLM-sourced rather than directly attacker-supplied, and the fetched body is never returned to the client (at most summarised into an 8-word caption — a very weak exfiltration channel). A 5-second timeout is already in place.

**Fix (optional hardening).** Before fetching: require `https:`, reject private/loopback/link-local hosts, and cap the response size. Low priority given the runtime constraints.

**Acceptance criteria.**
- [ ] `fetchArticle()` rejects non-`https:` and private/loopback hosts (if implemented).

## Quality findings

### Q-1 — Stage 3 full-text refinement is dead code (High, correctness)

**Location:** `src/worker.js:293` (`refineWithFullText`).

**Problem.** The Anthropic request is built with **`messages: []`** and the whole prompt placed in `system`. The Messages API rejects an empty messages array with a 400 (verified against the Claude API reference: "messages: at least one message is required"). So the call always throws, the `catch` returns the original articles, and **the entire full-text refinement feature is inert** in both the cron and on-demand paths. Worse, it first fetches the full HTML of up to three article URLs (`fetchArticle`) on every refresh, only to discard the result — latency and egress for zero effect. By contrast, `judgeAndCurate` (`src/worker.js:463`) passes a proper `user` message and works.

**Decision:** keep the feature and fix it — the full-text refinement is wanted for verdict quality, not removed.

**Fix.** Give the request a `user` message. The minimal change is to move the per-article context into a user turn and keep only the instructions in `system` (mirroring how `judgeAndCurate` already works):

```js
const r = await fetch("https://api.anthropic.com/v1/messages", {
  method: "POST",
  headers: { /* unchanged */ },
  body: JSON.stringify({
    model: JUDGE_MODEL,
    max_tokens: 400,
    system: refinementPrompt + '\n\nRespond with ONLY minified JSON: {"refined":[{"index":number,"verdict":"probing"|"fixating"|"noting","caption":string}]}',
    messages: [
      { role: "user", content: articleContexts },
    ],
  }),
});
```

The existing `try/catch` and `VERDICTS` validation around the response stay as-is — they correctly handle a malformed reply; the bug was only that the request never succeeded. Keep the graceful fallback to the original articles for the genuine-failure case (network error, bad JSON).

**Acceptance criteria.**
- [ ] The refinement request succeeds and actually returns refined verdicts/captions (no longer 400s on every call).
- [ ] Article HTML is still only fetched as input to a refinement that now runs for real.
- [ ] A unit test asserts the refinement path applies a changed verdict on a valid response, and falls back to the originals on a failed/garbled response (see Q-2).

### Q-2 — No automated tests despite the coverage mandate (High, process)

**Location:** `package.json` (no `test` script); no test files for `src/` or `public/`.

**Problem.** The committed `CLAUDE.md` mandates TDD and 95%+ coverage, and `SPECIFICATIONS/ORIGINAL_IDEA/project-outline.md` §126 explicitly decided to "write tests for both Worker logic and client-side rendering." Phase 5 "testing" was entirely manual checklists. By the project's own definition of done, the work is incomplete — and the Q-1 bug is exactly what one Worker unit test would have caught.

**Fix.** Add a test runner (Vitest pairs well with Workers; `@cloudflare/vitest-pool-workers` for Worker-context tests) and cover at least:

- **Worker:** `judgeAndCurate` parsing and verdict-allowlist enforcement; `handleCommentary` cache hit/miss/empty paths; `handleRefresh` auth (header present/absent/wrong — see S-3); the Q-1 refinement path (success and graceful-failure).
- **Front end:** that HTML metacharacters in `title`/`outlet`/`caption` render as inert text (locks in S-1). DOM-render functions are testable under jsdom/happy-dom if the script is extracted (which S-2 wants anyway).

Add a `test` script and a `test:coverage` script to `package.json`.

**Acceptance criteria.**
- [ ] `npm test` runs and passes.
- [ ] Coverage approaches the 95% lines / 90% branches target for `worker.js`.
- [ ] A regression test exists for S-1 (escaping) and Q-1 (refinement behaviour).

### Q-3 — Smaller fixes (Low)

**Q-3a — Article headlines are not clickable.** `renderArticles` has each article's `url` (the Worker even fetches it server-side) but never links the title. Wrap the title in an `<a href>` (with `rel="noopener noreferrer"` and `target="_blank"`) so readers can reach the source. Build the link via DOM/`textContent` per S-1 — do not interpolate `url` into `innerHTML`.

**Q-3b — README typo.** `README.md` reads "mainlyb really just a test" — fix to "mainly really just a test" (or reword).

**Acceptance criteria.**
- [ ] Article titles link to their source, opened safely.
- [ ] README typo corrected.
